    pb_sign_in_rule (Resource)

    Example Usage

    resource "pb_sign_in_rule" "example" {
  name   = "Allow corporate sign-in"
  mode   = "active"
  action = "allow"
}

    Argument Reference

    The following arguments are required:

    • action (String) Provide action to describe this rule
    • mode (String) The mode of the rule. The 'monitor' mode is not supported for this rule type.
    • name (String) The name or title of the rule.

    The following arguments are optional:

    • configuration_version (String) Configuration version to retrieve (draft, active, or version number)
    • description (String) The detailed description of the rule.
    • positioning (Attributes) Optional placement directive for the entity. On create, when omitted, the entity is placed at the top of the stack. On update, when omitted, the entity's current position is preserved. (see below for nested schema)
    • scope (Attributes) Provide scope to describe this rule (see below for nested schema)

    Attributes Reference

    In addition to the arguments above, the following attributes are exported:

    • id (String) The unique identifier of the created resource (PULID, 29 chars).
    • metadata (Attributes) (see below for nested schema)
    • priority (Number) Order position of the rule in the list
    • section (Attributes) Section this rule belongs to (null if standalone) (see below for nested schema)

    Nested Schema for positioning

    Required:

    • target (Attributes) (see below for nested schema)

    Nested Schema for positioning.target

    Required:

    • position (String) Where to place the entity: top (first), bottom (last), before (above anchor), after (below anchor). The before and after values require the anchor field. The top and bottom values must not include an anchor field — providing an anchor with top or bottom is a validation error.

    Optional:

    • anchor (Attributes) (see below for nested schema)
    • section_id (String) The section the rule should belong to. Only valid for rule POST and PATCH; must be null/omitted for section POST and PATCH (sections cannot be nested). The anchor (if any) must reside in this same container.

    Nested Schema for positioning.target.anchor

    Required:

    • id (String) The entity ID of the anchor item.
    • type_ (String)

    Nested Schema for scope

    Optional:

    • device_groups (Attributes) The device groups the rule applies to. (see below for nested schema)
    • locations (Attributes) The locations the rule applies to. (see below for nested schema)
    • private_ips (Attributes) The private IP addresses the rule applies to. (see below for nested schema)
    • public_ips (Attributes) The public IP addresses the rule applies to. (see below for nested schema)
    • users (Attributes) The users or user groups the rule applies to. (see below for nested schema)

    Nested Schema for scope.device_groups

    Optional:

    • device_groups (List of String)
    • is_any (Boolean) Flag indicating if any device groups are included in the scope. If isAny is set to true, no other fields should be present, and all device groups will be included in the scope. If isAny is set to false, the final scope must contain at least one device group.

    Nested Schema for scope.locations

    Optional:

    • is_any (Boolean) Flag indicating if any locations are included in the scope. If isAny is set to true, no other fields should be present, and all locations will be included in the scope. If isAny is set to false, the final scope must contain at least one location.
    • locations (List of String) Geographical locations to which the rule applies.

    Nested Schema for scope.private_ips

    Optional:

    • is_any (Boolean) Flag indicating if any private IP addresses are included in the scope. If isAny is set to true, no other fields should be present, and all private IP addresses will be included in the scope. If isAny is set to false, the final scope must contain at least one private IP.
    • private_ips (List of String) Private IP addresses affected by the rule.

    Nested Schema for scope.public_ips

    Optional:

    • is_any (Boolean) Flag indicating if any public IP addresses are included in the scope. If isAny is set to true, no other fields should be present, and all public IP addresses will be included in the scope. If isAny is set to false, the final scope must contain at least one public IP.
    • public_ips (List of String) Public IP addresses affected by the rule.

    Nested Schema for scope.users

    Optional:

    • is_any (Boolean) Flag indicating if any users or user groups are included in the scope. If isAny is set to true, no other fields should be present, and all users and user groups will be included in the scope. If isAny is set to false, the final scope must contain at least one user or user group.
    • user_groups (List of String)
    • users (List of String)

    Nested Schema for metadata

    Read-Only:

    • configuration_version (Attributes) Configuration version information (see below for nested schema)
    • created_by (String) Email or identifier of the admin who created the rule
    • created_time (String) Timestamp when the rule was created
    • last_updated_by (String) Email or identifier of the admin who last updated the rule
    • last_updated_time (String) Timestamp when the rule was last updated

    Nested Schema for metadata.configuration_version

    Read-Only:

    • id (String) Configuration version identifier
    • number (Number) Configuration version number
    • status (String) Configuration version status

    Nested Schema for section

    Read-Only:

    • id (String) ID of the section
    • name (String) Name of the section
