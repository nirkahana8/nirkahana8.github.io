    pb_access_and_data_rule (Resource)

    Example Usage

    resource "pb_access_and_data_rule" "example" {
  name = "Restrict uploads to sanctioned apps"
  mode = "active"

  applications = {
    # Application scope for the rule (see the Schema section below).
  }
  access = {
    # Access control configuration (see the Schema section below).
  }
  tracking = {
    # Tracking / logging configuration (see the Schema section below).
  }
}

    Schema

    Required

    • access (Attributes) Access control configuration for an access-and-data rule.

    Note: on PATCH, the access block uses replace semantics — clients must re-send the full object on every update. Any field omitted is cleared (with the exception of internal fields not exposed by this API, such as authMethod and enforceRedirect, which are preserved from the existing rule). (see below for nested schema) - applications (Attributes) Applications scope for creating an access-and-data rule, categorized by application type. (see below for nested schema) - mode (String) The mode of the rule. - name (String) The name or title of the rule. - tracking (Attributes) Tracking and logging configuration for the rule (see below for nested schema)

    Optional

    • configuration_version (String) Configuration version to retrieve (draft, active, or version number)
    • data_controls (Attributes) Data controls for access-and-data rules.

    Each data control supports a specific set of data-profile content types (no profile, on-device scanning, or cloud-assisted scanning). When a rule links a data profile (an EDLP dlpProfileId) or uses inline content, controls that are not supported for the resulting content type are rejected with HTTP 400. Set bypassDlpProfileValidation: true to suppress content-agnostic compatibility rejections only — those controls save successfully but become runtime no-ops. Controls that are entirely unsupported for the resulting content type are always rejected, even when bypass is enabled. (see below for nested schema) - description (String) The detailed description of the rule. - login_controls (Attributes) Login controls for access-and-data rules. (see below for nested schema) - positioning (Attributes) Optional placement directive for the entity. On create, when omitted, the entity is placed at the top of the stack. On update, when omitted, the entity's current position is preserved. (see below for nested schema) - scope (Attributes) Provide scope to describe this rule (see below for nested schema)

    Read-Only

    • id (String) The unique identifier of the created resource (PULID, 29 chars).
    • metadata (Attributes) (see below for nested schema)
    • priority (Number) Order position of the rule in the list
    • section (Attributes) Section this rule belongs to (null if standalone) (see below for nested schema)

    Nested Schema for access

    Required:

    • action (String) Top-level web-access decision.

    Optional:

    • address_bar_label (String) Label shown in the browser address bar for matched pages.
    • authentication_duration_minutes (String) Duration in minutes (as a string) for a timeframe window (e.g. user-prompt bypass window or authentication session validity). Allowed values: 0 (single use / every time), 10 (10m), 60 (1h), 240 (4h), 540 (9h), 720 (12h), 1440 (24h), 4320 (3d), 10080 (7d), 20160 (14d), 43200 (30d), 86400 (60d), 129600 (90d).
    • custom_dialog_text (Attributes) Custom dialog text shown to the user. (see below for nested schema)
    • custom_label (Attributes) Custom address-bar label configuration. Required when addressBarLabel is 'custom'. (see below for nested schema)
    • mfa_required (Boolean) Whether the user must complete an MFA challenge before the control's restricted action is allowed. The MFA method (pinCode / identityProvider) is configured globally on the Browser Security Authentication Factor control and is not set per-rule.
    • open_in_prisma_browser (Attributes) Configuration for the Extension to Prisma Browser handoff. (see below for nested schema)
    • user_prompt (Attributes) How an end user is prompted before access is granted, and how long their decision is remembered. (see below for nested schema)

    Required:

    Custom Apps: * Protocol: Defaults to *:// if omitted. * Subdomain: Defaults to www logic (hidden). * Path: Appends /* if no path is specified.

    Private Apps: * Protocol: Defaults to https://. Wildcards (*) are NOT allowed. * Subdomain: No automatic www handling; www is treated as a unique prefix. * Domain: Supports IPv4 and Port numbers (e.g., :8080). * Path: Automatically adds a trailing * to all paths.

    Nested Schema for data_controls

    Optional:

    • bypass_dlp_profile_validation (Boolean) Bypass the content-agnostic data-profile compatibility check for controls in this object. When false (default), controls incompatible with the configured dlpProfileId are rejected with HTTP 400. When true, the rule saves and content-agnostic incompatible controls become runtime no-ops. Controls entirely unsupported for the resulting content type are always rejected, even when this is true.
    • camera (Attributes) Control the ability to use camera in web pages. (see below for nested schema)
    • clipboard (Attributes) Control cut, copy and paste. (see below for nested schema)
    • custom_http_header (Attributes) Set a custom HTTP header to identify HTTP/S requests. (see below for nested schema)
    • developer_tools_on_web_pages (Attributes) Control DevTools to avoid potential data leaks, inspecting page content, and running unauthorized code in the console. (see below for nested schema)
    • dlp_profile (Attributes) Expanded reference to an Enterprise DLP (EDLP) data profile. (see below for nested schema)
    • dlp_profile_id (String) Enterprise DLP (EDLP) data profile id (the same value returned by EDLP GET /v2/api/data-profiles). Write-only on POST/PATCH; reads return dlpProfile instead. Send null on PATCH to unlink the rule from any EDLP profile.
    • file_protection (Attributes) Control how files enter and leave the rule's applications. Configure fileDownload and fileUpload independently — each direction owns its action, filters, prompt, MFA setting, and custom dialog text. Replaces the legacy fileDownload and fileUpload top-level controls. (see below for nested schema)
    • gen_aiprompt (Attributes) A simple control with an allow or block action. Re-usable by any data control that only needs a binary allow/block decision. (see below for nested schema)
    • live_page_scanning (Attributes) Prevent Exposure to malicious pages using real-time page scanning, powered by Prisma Browser's Advanced Web Protection. (see below for nested schema)
